# filmkit (eggricesoy/filmkit) — Browser PTP-over-WebUSB RAW Converter and Preset Manager: Protocol Reference for Frameport

**Source project:** eggricesoy/filmkit | **Language:** TypeScript | **Transport:** PTP over WebUSB (USB-only; no PTP-IP, no BLE) | **Primary hardware:** Fujifilm X100VI (confirmed, 2026-03)

filmkit is a browser-based Fujifilm X-series preset manager and in-camera RAW (RAF) converter. It communicates with the camera over a direct USB connection using the WebUSB API and speaks full PTP at the container layer. For Frameport, filmkit is a high-confidence reference for: the PTP container wire format (12-byte header, all field offsets, endianness), the complete Fujifilm vendor device property map (0xD001–0xD1A5), the film simulation numeric code table, the C1–C7 preset slot read/write protocol including Wireshark-verified write ordering, the D185 native profile structure (625 bytes, X100VI confirmed), the HighIsoNR non-linear encoding table, grain-effect encoding duality, and the Fujifilm-vendor-opcode RAF upload flow (0x900C / 0x900D). Because PTP containers are transport-agnostic above the framing layer, every container-level and property-level fact documented here applies without modification to Frameport's PTP-IP Wi-Fi path in `fuji-ptpip` and `fuji-core`. filmkit does not implement PTP-IP, BLE, or live-view; those layers must be sourced elsewhere.

## 4. Standout Findings

1. **Two incompatible D185 profile formats exist.** The camera returns a 625-byte native format (confirmed X100VI, 2026-03); rawji documents a 632-byte write format. Frameport must implement the 625-byte native format. The field index table in `d185.ts` (NativeIdx) is the confirmed mapping for the native format.

2. **HighIsoNR (D1A1 / profile index 20) uses a proprietary NON-LINEAR encoding, not ×10.** The complete lookup table is: −4→0x8000, −3→0x7000, −2→0x4000, −1→0x3000, 0→0x2000, +1→0x1000, +2→0x0000, +3→0x6000, +4→0x5000. Critically, 0x8000 means −4 NR here, whereas for all other tone fields 0x8000 is the "use EXIF default" sentinel — context-dependent meaning of the same bit pattern.

3. **Grain effect has two different wire encodings that must not be confused.** The D185 profile uses a byte-packed `uint16` (high byte = size 0/1, low byte = strength 0/2/3). The preset properties (D195) use a flat 1-indexed enum (1=Off, 2=WeakSmall, 3=StrongSmall, 4=WeakLarge, 5=StrongLarge).

4. **Effect properties (ColorChrome D196, ColorChromeFxBlue D197, SmoothSkin D198) use 1-indexed encoding in both profile and preset wire formats** (1=Off, 2=Weak, 3=Strong), not 0-indexed. Writing 0 to these will likely be rejected or mis-interpreted by the camera.

5. **Preset property write order matters and was confirmed via Wireshark capture of the official X RAW Studio app.** D19C (color temperature in Kelvin) must be written immediately after D199 (WB mode), and only when WB mode is 0x8007 (ColorTemp). WB shift registers D19A/D19B come after D19C. Violating this order may cause silent write failures.

6. **Stale PTP sessions (SessionAlreadyOpen = 0x201E) require a full transport reset (not just CloseSession) before retrying OpenSession.** Transaction IDs must be reset to 0 after transport reset. Without this, the camera and host will have mismatched transaction ID expectations.

7. **RAF upload uses Fujifilm VENDOR opcodes 0x900C (SendObjectInfo) and 0x900D (SendObject2), not standard PTP 0x100C/0x100D.** The ObjectInfo payload must specify ObjectFormat=0xF802 for RAF; the filename must be exactly `FUP_FILE.dat`. A wrong format causes silent failure — no error response, conversion simply does not work.

8. **The WB mode property (D199 in presets) is stored as `uint16` but the generic property reader returns it as a signed `int16`.** Values like 0x8001 (Fluorescent1) decode as negative signed `int16` (−32767). Callers must mask with 0xFFFF to recover the original `uint16` WB code before matching against the WbMode enum.

## 5. Caveats / IP and License Risk

- **Single hardware body.** filmkit is tested exclusively on the Fujifilm X100VI. The film simulation codes (0x01–0x14) are noted in the source as "from rawji, may need adjustment for X-Processor 5". Property code availability varies by camera model — check DeviceInfo.properties before reading/writing any property.

- **D185 format confirmed on X100VI only.** The 625-byte D185 native profile format and NativeIdx field mapping were confirmed via X100VI testing in March 2026. Other X-series models (X-T5, X-H2) likely share the same format based on the same X-Processor 5, but this has not been independently verified in filmkit. The 632-byte rawji format applies to different cameras or firmware.

- **HighIsoNR encoding is empirical, not derived from a formula.** The NR_ENCODE/NR_DECODE tables were reverse-engineered from Wireshark captures of Fujifilm X RAW Studio. The pattern does not follow any mathematical formula. Future firmware versions could change it.

- **Two property codes have unknown semantics.** D191 (observed always 0) and D1A5 (observed always 7) have unknown semantics. Their observed defaults from camera scans are recorded, but the camera may interpret them differently on models or firmware versions not tested.

- **filmkit has no PTP-IP implementation.** All transport facts here are PTP-over-USB/WebUSB. For Frameport's Wi-Fi PTP-IP path, the PTP container format and property codes are the same, but the framing layer (PTP-IP TCP init/command/event socket protocol) must be sourced from other references (libgphoto2, rawji, the CIPA DC-005 PTP-IP standard).

- **License review required.** filmkit is released under a license that should be reviewed before any implementation decisions. The numeric protocol constants and packet layouts extracted here are interoperability facts from the ISO 15740 PTP standard and from clean-room reverse engineering of camera behavior — they are not copyrightable. However, any Frameport implementation must be written from scratch in Rust, not ported from the TypeScript source.

- **Preset property mapping is single-body confirmed.** The confirmed mapping (D18C–D1A5) was derived by cross-referencing 7 camera presets on X100VI hardware with known settings. The mapping comment in `constants.ts` explicitly notes it was "Confirmed via cross-referencing 7 camera presets (X100VI, 2026-03)" which provides strong confidence but is still based on a single camera body.

- **Wireshark captures not published.** filmkit explicitly does not accept pull requests and is a read-only reference. The Wireshark captures used to derive encoding tables are not published, only the derived lookup tables and behavioral constraints.

- **MUST-NOT-EMBED.** No Fujifilm proprietary keys, tokens, cloud endpoints, credentials, or extracted firmware binaries are present in filmkit's codebase. No such assets must be embedded in Frameport regardless.
